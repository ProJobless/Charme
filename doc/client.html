TEXT: Key Directory

Info

Each user has a key directory. It contains the public keys of other users and also the symmetric edgekeys, which can be used for faster encryption.

The mongoDB collection is called keydirectory. Each entry contains the following JSON fields:

  • owner - The owner of the key directory
  • userId - the userId of the public key owner
  • pubKeyRevision - Revision of the public key. It is needed for decrypting the rsaEncEdgekey
  • fkrevision - fastkey revision
  • fkEncEdgekey - The edgekey for the key directory owner encrypted with the symmetric fastkey
  • rsaEncEdgekey - The edgekey for the person in the key directory
  • value - The public key encrypted

TEXT: Login Process

Info

The login() function is located in routes.js. After having read out username and password, we send a user_login we either decrypt the passphrase with the password first. Then we need to get the salt value for the user by sending the request reg_salt_get to the server. We hash the password with this value and send it to the server by calling user_login. Then we get the encrypted keyring which contains all private and public keys of the user and decrypt it with the passphrase.

TEXT: Sending Messages

Info

When sending messages, the files play a significant role:

  • talks.js - UI Logic
  • req.php - The server component
  • hydra.php - message distribution via Gearman

The message answer action is triggered by $('#but_instantsend').click. First we get the key for the current conversation and define msgKey with it. The Message is encrypted with that key and send to the server with message_distribute_answer. The request is then parsed on the server in req.php with action message_receive.

TEXT: Encrypting Messages

Info

Messages are encrypted with public key cryptography. When adding a users key to a key directory, an asymmetric AES edgekey is established from user A to user B. This is done for performance reasons. The edgekeys are obtained via request action edgekeys_byUserIds.

Graph
